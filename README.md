# 数字电源框架模板说明

## 项目简介

这套代码是一套面向数字电源开发的框架模板，核心目标不是绑定某一个具体硬件平台，也不是绑定某一种单一拓扑，而是沉淀出一套可复用、可扩展、可移植的数字电源软件架构。

这套框架的核心优势是：

- 软硬件解耦
- 模块化组织
- 统一调度框架
- 基础算法、基础通信和调试能力齐全
- 同一套 code 可以同时兼容不同 MCU 平台和 PLECS 平台

也就是说，这套框架的重点不是去强调某个 MCU 平台本身的能力，也不是去强调某个仿真平台本身的能力，而是强调代码架构本身的通用性。对于数字电源项目来说，只要底层接口完成适配，同一套控制与业务代码就可以在不同硬件平台和 PLECS 平台之间复用，这就是这套框架最大的价值。

---

## 这套框架的核心优势

### 1. 软硬件完全解耦

这套框架最重要的优势，就是把控制逻辑和硬件实现尽量分开。

在功率控制模块中，算法层并不直接依赖某个具体芯片寄存器，也不直接绑定某一版板级资源，而是通过 HAL 和接口层来获取：

- 采样量
- PWM 输出接口
- GPIO/继电器控制接口
- 运行许可和状态输入

这样做的直接价值是：

- 更换 MCU 平台时，不需要重写控制主体
- 更换功率板时，不需要推翻整套控制代码
- 更换采样链路或驱动方式时，主要修改接口映射层
- 控制算法可以脱离具体硬件长期积累

这意味着这套代码并不是“某一块板子的专用代码”，而是一套真正的平台型代码。

### 2. 模块化程度高，适合数字电源工程扩展

在 `code/ctrl` 下，每个功率控制模块基本按统一结构组织：

- `*_cfg`：配置与设定值管理
- `*_hal`：硬件抽象层
- `*_ctrl`：控制算法主体
- `*_fsm`：运行状态机

这种组织方式的意义在于：

- 每个模块职责清晰
- 模块之间边界明确
- 新增功能时不容易破坏原有结构
- 同类拓扑可以快速复制和派生

对于数字电源项目来说，这种模块化非常重要。因为实际项目中常常不是只有一个控制环，而是多个功率模块并存，并且还会随着项目演进不断增加新模块、新保护、新联动逻辑。如果没有模块化结构，代码很快就会变成强耦合、难以维护的工程。

### 3. 一套 code 同时兼容 MCU 平台和 PLECS 平台

这套框架的另一个核心优势，是平台兼容性建立在统一代码架构之上。

它不是分别维护一套 MCU 代码和一套 PLECS 代码，而是通过平台抽象，把平台相关部分收敛到接口层和平台适配层。这样一来：

- 控制算法主体可以不变
- 业务逻辑主体可以不变
- 状态机主体可以不变
- 调度框架主体可以不变
- 需要变化的只是底层平台适配和硬件映射

因此，这套代码的优势不在于“支持某个 MCU”或者“支持 PLECS 仿真”，而在于它本身就是平台无关的。无论最终落到哪种 MCU，还是落到 PLECS 平台做验证，核心代码都可以保持一致。

这对于数字电源开发非常关键，因为它大幅降低了平台切换带来的重复劳动，也保证了控制逻辑在不同环境下的一致性。

### 4. 低耦合设计带来的工程价值

这套框架除了结构清晰，更重要的是模块之间的耦合度被刻意压低了。

低耦合带来的优势主要体现在：

- 更容易复用
  同一个控制模块、算法库、通信模块可以跨项目继续使用。

- 更容易移植
  平台变化时，主要修改底层适配，不必大范围改上层逻辑。

- 更容易调试
  可以快速区分问题来自算法、采样、PWM、状态机还是通信。

- 更容易扩展
  新增功率模块或新增功能时，只需按既有接口接入。

- 更适合多人协作
  控制、接口、通信、调试、业务流程可以并行开发，互相影响较小。

- 更利于长期维护
  后续硬件改版、参数调整、策略升级时，不容易出现“改一处牵一片”的问题。

从工程化角度看，低耦合意味着这套代码不会被单一项目绑死，而是可以持续沉淀为数字电源平台代码。

## 框架核心机制

### 1. `section` 统一调度框架

这套代码的核心骨架是 `code/section`。

它通过自动注册机制，把初始化、周期任务、中断任务、状态机、链路处理等统一纳入同一套调度框架。模块不需要把所有内容手工集中登记到一个总表中，而是可以在模块内部直接完成注册。

主要注册机制包括：

- `REG_INIT()`：注册初始化函数
- `REG_TASK()` / `REG_TASK_MS()`：注册周期任务
- `REG_INTERRUPT()`：注册中断阶段任务
- `REG_FSM()`：注册状态机驱动任务
- `REG_LINK()`：注册链路输入处理
- `REG_COMM()`：注册通信命令处理

这种设计非常适合数字电源，因为数字电源天然就是多时基系统：

- 高速控制在中断里执行
- 快速保护和测量处理在快速任务里执行
- 参数管理、状态更新、通信上报在慢速任务里执行

而 `section` 框架正好把这些不同节拍、不同职责的代码统一组织起来。

### 2. 功率控制模块的解耦结构

控制模块通常按以下结构拆分：

- `cfg`：负责参数、设定值、上下限和运行许可管理
- `hal`：负责对接平台输入输出接口
- `ctrl`：负责控制算法计算
- `fsm`：负责状态流转和运行流程

这种拆分使得：

- 算法只处理控制逻辑
- HAL 只负责硬件映射
- FSM 只负责流程切换
- 配置层只负责参数组织

从而让整个功率模块既清晰又稳定。

### 3. 基础算法库

`code/lib` 提供了数字电源常用的基础算法模块，例如：

- PI / PID / PR
- SOGI / FLL
- Notch
- RMS 计算
- MPPT
- 滞回比较与辅助线性模块

这些基础库的存在，使得框架具备了实际数字电源开发常用的算法底座。

### 4. 基础通信与调试机制

框架内置了通信协议层和 shell 调试机制。

在 `code/comm` 中已经包含：

- 基础帧协议
- CRC16
- 状态机解析
- 命令注册与分发
- 参数读写
- 数据上报
- Shell 命令与变量访问

这使得整套框架不仅能跑控制，还具备数字电源开发所需的调参、联调、在线观察和通信扩展能力。

## 代码总体架构

### 目录说明

```text
base
├─ code
│  ├─ app         上层业务逻辑、保护、流程控制、数据交互
│  ├─ comm        基础通信协议与 shell 调试框架
│  ├─ ctrl        功率控制模块
│  ├─ dbg         调试与波形输出
│  ├─ interface   硬件接口封装
│  ├─ lib         基础算法库与公共工具
│  └─ section     框架核心：自动注册与统一调度
├─ gd32g553c      平台落地示例
└─ plecs          平台落地示例
```

这里需要强调的是：

- `gd32g553c` 和 `plecs` 不是这套框架的本体优势
- 真正的优势是 `code` 这套架构本身
- 平台目录只是这套架构落地到不同平台时的适配层示例

### 当前平台落地说明

当前仓库中已经包含一套 PLECS 平台落地方式，用于让同一套框架代码在 PLECS 环境中直接运行。

当前使用方式为：

- PLECS 版本：`4.7.6`
- 编译器：`MinGW64`
- 生成形式：编译为 `DLL`，供 PLECS 工程加载运行
- 安装好环境后，直接运行compile.bat脚本

### 分层理解

#### 1. 框架层

对应 `code/section`

职责：

- 自动注册
- 初始化组织
- 周期任务调度
- 中断任务组织
- 状态机驱动
- 链路调度

#### 2. 公共能力层

对应 `code/lib`、`code/comm`、`code/dbg`

职责：

- 基础控制算法
- 通信协议
- 参数调试
- 调试输出
- 公共工具能力

#### 3. 接口层

对应 `code/interface`

职责：

- 采样接口封装
- PWM 接口封装
- GPIO/通信接口封装
- 屏蔽平台差异

#### 4. 功率控制层

对应 `code/ctrl`

目前主要包含：

- `pfc`
- `inv`
- `bb`

#### 5. 应用层

对应 `code/app`

职责：

- 上层流程
- 保护逻辑
- 模块联动
- 业务组织

## 总结

这套代码本质上是一套面向数字电源的软件框架模板，核心优势可以概括为三点：

- 软硬件解耦
- 模块化、低耦合
- 一套 code 同时兼容 MCU 平台和 PLECS 平台

它通过 `section` 统一调度框架，把初始化、任务、中断、状态机、通信和调试机制组织成一个可扩展的整体；再通过 `cfg + hal + ctrl + fsm` 的模块拆分方式，把功率控制模块做成可复用、可移植、可扩展的独立单元。

因此，这套框架真正的价值不在于某个具体平台，而在于它提供了一套可以长期沉淀、跨项目复用、跨平台兼容的数字电源软件架构。
