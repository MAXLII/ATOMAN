# 数字电源框架模板说明

## 项目简介

这套代码是一套面向数字电源开发的嵌入式框架模板，核心目标不是直接交付某一个单一产品功能，而是提供一套可复用、可裁剪、可移植的代码组织方式，让 PFC、LLC、逆变、Buck-Boost 等功率控制模块能够在统一框架下快速搭建、调试和移植。

项目当前以 `GD32G553` 为主控平台进行落地，同时保留了 `PLECS` 联调/验证环境，因此它既适合作为 MCU 固件模板，也适合作为控制算法和硬件接口联合验证的基础工程。

这套框架的重点不在于“把所有功能写死”，而在于提供一套适合数字电源的软件架构：

- 功率控制模块可以尽量完全解耦
- 初始化、周期任务、中断、状态机统一纳入 `section` 调度框架
- 控制算法、硬件抽象、上层应用分层清晰
- 基础库、通信、调试接口都有现成实现
- 便于后续扩展到不同拓扑、不同功率等级、不同硬件平台

## 为什么这套框架适合数字电源开发

数字电源软件和普通 MCU 应用最大的区别，是它天然具有以下特点：

- 既有高速闭环控制，又有慢速管理任务
- 既有强实时中断逻辑，又有状态机与流程控制
- 既要关注算法本身，又要关注采样、PWM、继电器、保护等硬件协同
- 同一个项目里经常并存多个功率模块，例如 PFC、LLC、逆变、Buck-Boost
- 不同项目之间往往“控制思想类似、硬件接口不同”，非常适合模板化和平台化

这套框架正好针对这些特点进行了设计。

### 1. 功率控制模块可以做到高度解耦

在 `code/ctrl` 下，每个功率模块基本都按统一模式组织：

- `*_cfg`：配置与设定值管理
- `*_hal`：硬件抽象层，负责把控制器需要的测量量和执行接口接进来
- `*_ctrl`：控制算法主体
- `*_fsm`：运行流程和状态机

例如 `pfc`、`inv`、`bb` 都采用了类似结构。这样做的意义是：

- 算法层不直接依赖底层寄存器
- 控制器只关心“输入量、输出量、控制目标”
- 更换功率板、采样链路或 PWM 外设时，主要改 `hal` 层
- 更换控制参数和运行策略时，主要改 `cfg` 和 `fsm`
- 同类模块可以复制模板后快速派生新工程

这正是数字电源项目最需要的能力，因为实际开发中常常是“硬件版本变化快，控制框架必须稳定”。

### 2. `section` 调度方式很适合数字电源这种多时基系统

数字电源通常天然是多速率系统：

- 控制 ISR 运行在高速节拍
- 保护、测量处理运行在快速任务
- 通信、打印、状态管理运行在慢速任务

本工程用 `code/section/section.h` 和 `code/section/section.c` 建立了一套统一自动注册机制。模块不需要手工集中登记，只需要通过宏声明即可接入框架：

- `REG_INIT()`：注册初始化函数
- `REG_TASK()` / `REG_TASK_MS()`：注册周期任务
- `REG_INTERRUPT()`：注册中断阶段任务
- `REG_FSM()`：注册状态机驱动任务
- `REG_LINK()`：注册链路输入处理
- `REG_COMM()`：注册通信命令

主函数非常简洁：

```c
int main(void)
{
    systick_config();
    section_init();
    while (1)
    {
        run_task();
    }
}
```

这说明框架的运行模式很清晰：

1. 上电后通过 `section_init()` 自动扫描各类注册项
2. 构建初始化链表、任务链表、中断链表、链路链表
3. 执行所有初始化函数
4. 在主循环里统一调度周期任务

对于数字电源来说，这种方式有几个明显好处：

- 新增模块时，不需要去改一份庞大的“总调度表”
- 任务注册和模块代码写在一起，维护成本低
- 控制中断、慢速任务、状态机可以统一纳入框架
- 项目规模变大时，代码仍然容易组织

### 3. 控制代码与硬件接口分离，便于移植

框架中 `ctrl` 层和 `interface`/`bsp` 层职责分开：

- `code/ctrl` 负责控制算法和控制状态机
- `code/interface` 负责 ADC、PWM、GPIO、USART 等接口封装
- `gd32g553c/bsp` 负责芯片和板级初始化

例如在 `pfc_hal.c` 中，控制器依赖的是一组指针和函数接口，而不是某个具体寄存器：

- 电压、电流采样值
- PWM 使能/关闭
- PWM 设置函数
- 继电器控制与运行许可

这意味着：

- 控制算法可以独立演进
- 硬件平台迁移时修改范围可控
- 便于在 PLECS 和实机之间做一致性映射

这正是数字电源软件工程化的关键，不然控制代码很容易和某一版硬件强耦合，后期难以复用。

### 4. 状态机机制适合电源上电、预充、运行、保护等流程

数字电源除了控制环路，还必须有完善的运行流程管理，例如：

- 上电初始化
- 待机
- 软启动
- 继电器吸合
- 闭环运行
- 故障停机
- 故障恢复

本工程提供了表驱动状态机宏 `REG_FSM()`，并已用于多个模块，例如：

- `app/pfc/pfc_fsm.c`
- `app/llc/llc_fsm.c`
- `ctrl/pfc/pfc_fsm.c`
- `ctrl/inv/inv_fsm.c`
- `ctrl/bb/bb_fsm.c`

这样做的好处是：

- 运行流程清晰
- 状态切换逻辑集中
- 进入态、运行态、退出态职责明确
- 很适合电源系统这种“流程 + 控制”并存的软件形态

### 5. 已经具备常用的基础算法库

`code/lib` 中已经提供了较多数字电源常用基础算法和工具模块，例如：

- `pid` / `pid_inc`
- `pi_tustin`
- `pr`
- `notch`
- `sogi`
- `fll`
- `mppt`
- `cal_rms`
- `hys_cmp`
- `linear`
- `ac_loss_det`
- `time_share`
- `volt_mt`

这些库的意义在于：

- 可以直接支撑 PFC、逆变、Buck-Boost、MPPT 等控制需求
- 项目启动时不用从零搭基础算法
- 算法模块可以跨项目复用
- 有利于形成统一的控制实现风格

对于数字电源开发来说，基础库的沉淀价值非常高，因为真正耗时的往往不是“写一个 if-else”，而是把这些控制基础件长期稳定地维护好。

### 6. 已经具备基础通信和调试能力

框架内置了链路层、通信协议层和 shell/参数调试机制。

#### 链路层

`REG_LINK()` 用于注册一个输入链路，典型场景是串口 DMA 环形缓冲接收。链路层只负责“字节流分发”，不关心上层协议细节。

典型实现位于：

- `code/interface/usart.c`
- `code/interface/ac/comm_link.c`

在一个链路上，可以并行挂多个 handler，例如：

- `shell_run`
- `comm_run`

这意味着同一串口既能做人机调试，也能跑二进制通信协议。

#### 通信协议层

`code/comm/comm.c` 和 `code/comm/comm.h` 实现了基础通信协议框架，包含：

- 帧格式定义
- CRC16 校验
- 状态机解析
- 命令注册
- 链路间路由
- 数据发送封装

通过 `REG_COMM()` 可以把命令处理函数直接注册到协议框架中，这种方式非常适合数字电源中常见的：

- 参数读写
- 状态上报
- 上位机调试
- 模块间转发
- BMS/从机/面板通信扩展

#### Shell 与参数调试

`code/comm/shell.c` / `shell.h` 提供了变量和命令注册能力：

- `REG_SHELL_VAR()`
- `REG_SHELL_CMD()`

可以支持：

- 在线查看变量
- 在线修改参数
- 自动打印状态
- 波形参数输出
- 调试命令执行

这对数字电源调试非常重要，因为控制参数、保护阈值、状态量通常都需要在联调阶段频繁观察和调整。

### 7. 兼顾实机运行与仿真验证

工程根目录同时保留：

- `gd32g553c/`：GD32 实机工程
- `plecs/`：PLECS 联调/仿真工程

并且 `section/platform.h` 针对 `IS_PLECS` 和 MCU 平台做了区分。这样做的价值是：

- 同一套框架可用于控制算法早期验证
- 有利于从仿真过渡到实机
- 有利于减少“仿真一套、固件一套”的割裂

对于数字电源项目，这种一致性非常有价值。

## 代码总体架构

### 目录说明

```text
base
├─ code
│  ├─ app         上层业务逻辑、模块流程、保护、升级、数据交互
│  ├─ comm        通信协议与 shell 调试框架
│  ├─ ctrl        功率控制模块，按拓扑分目录组织
│  ├─ dbg         调试与波形输出
│  ├─ interface   外设接口封装，如 ADC、PWM、GPIO、USART
│  ├─ lib         基础算法库与公共工具
│  └─ section     框架核心：自动注册、调度、状态机、链路
├─ gd32g553c      MCU 工程、启动文件、BSP、编译脚本
└─ plecs          PLECS 仿真与验证工程
```

### 分层理解

可以把整套代码理解为下面这几层：

#### 1. 框架层

对应 `code/section`

职责：

- 管理自动注册
- 统一初始化
- 周期任务调度
- 中断任务组织
- 状态机驱动
- 链路分发

这是整个工程的“骨架”。

#### 2. 公共能力层

对应 `code/lib`、`code/comm`、`code/dbg`

职责：

- 提供通用控制算法
- 提供通信协议
- 提供 shell 和调试输出
- 提供性能统计和基础工具

这是整个工程的“公共底座”。

#### 3. 硬件接口层

对应 `code/interface`、`gd32g553c/bsp`

职责：

- 对接 ADC、PWM、GPIO、USART、DMA
- 屏蔽底层芯片差异
- 为上层提供统一接口

这是算法与硬件之间的“隔离层”。

#### 4. 功率控制层

对应 `code/ctrl`

目前可以看到的主要模块包括：

- `pfc`
- `inv`
- `bb`

这层负责真正的控制算法、控制状态机和设定值管理，是数字电源项目的核心。

#### 5. 应用层

对应 `code/app`

职责：

- 上电流程
- 保护策略
- 数据交互
- 模块联动
- 特定业务逻辑

这层把控制层和整机行为连接起来。

## 这套模板最有价值的几个设计点

### 1. 以 `section` 为中心，实现“模块自带注册”

模块写完以后，初始化、任务、中断、状态机都可以在本模块内部直接声明，减少集中式维护成本。

这非常适合数字电源项目，因为一个成熟系统往往会不断加入：

- 新控制环
- 新保护逻辑
- 新调试命令
- 新通信命令
- 新业务流程

如果都集中到一个总文件里维护，工程很快会失控；而 `section` 机制能明显改善这一点。

### 2. 功率控制模块与硬件接口分离

控制器通过 HAL 拿测量值和执行接口，这使得：

- 控制算法可复用
- 硬件变化不至于冲击上层逻辑
- 控制仿真和实机接口更容易对齐

### 3. 低耦合设计带来的直接优势

这套框架除了“结构清晰”，更重要的是模块之间的耦合度被刻意压低了。  
低耦合对于数字电源项目不是锦上添花，而是决定项目能否持续复用和演进的关键。

它的主要优势体现在以下几个方面：

- 更容易复用  
  某个控制模块、通信模块、基础算法库写好以后，可以较少依赖其他业务代码，直接迁移到新项目中继续使用。

- 更容易移植  
  当主控芯片、驱动方式、采样通道或功率板变化时，通常只需要调整接口层或 HAL 映射，而不必大范围改动控制主体。

- 更容易调试和定位问题  
  控制异常时，可以更快区分问题究竟来自采样、PWM、状态机、通信，还是控制算法本身，避免所有问题混在一起。

- 更容易扩展功能  
  新增 PFC、LLC、逆变、Buck-Boost 的某个子模块时，不需要破坏已有结构，只要按现有接口和注册方式接入即可。

- 更适合多人协作  
  不同开发者可以分别负责控制算法、硬件接口、通信调试、上层流程，各自修改边界比较清晰，减少互相影响。

- 更利于长期维护  
  数字电源项目通常会经历参数调整、硬件改版、控制策略升级。低耦合结构能明显降低后期维护成本，避免“改一处、牵一片”。

从工程实践角度看，低耦合意味着这套代码不容易被单一项目绑死，而是可以逐步沉淀为真正的数字电源平台代码。

### 4. 既有控制算法框架，也有工程化基础设施

很多“算法工程”只有控制代码，没有完整的工程基础设施；而这套模板已经把：

- 调度
- 通信
- 调试
- 状态机
- 基础算法库

一起打包好了，因此更适合作为长期复用的项目母版。

### 5. 带有明显的可移植思路

`ctrl` 目录下的 checklist 文件也说明了这套代码不是写死在单一项目上的，而是有意识地把移植步骤、接口映射、参数校核整理出来，便于在新平台、新硬件、新拓扑上继续复用。

## 建议的使用方式

如果把这套代码作为新的数字电源项目模板，建议按下面的思路使用：

1. 先保留 `section + lib + comm + interface` 作为公共底座
2. 按目标拓扑复制或裁剪 `ctrl` 中对应模块
3. 在 `*_hal` 中完成采样量、PWM、继电器等接口映射
4. 在 `*_cfg` 中配置设定值、参数和上下限
5. 在 `*_fsm` 中整理启动、运行、保护和退出流程
6. 用 shell 和通信模块完成调参与联调
7. 结合 `plecs/` 先验证算法，再切换到实机

## 总结

这套代码本质上是一套面向数字电源的软件框架模板，而不是单纯的“功能集合”。

它之所以适合数字电源开发，关键在于它同时解决了几个核心问题：

- 用 `section` 机制解决了多模块、多任务、多时基系统的组织问题
- 用 `cfg + hal + ctrl + fsm` 结构解决了功率控制模块的解耦问题
- 用 `lib` 解决了常用控制算法复用问题
- 用 `comm + shell + dbg` 解决了调试、通信和参数管理问题
- 用 `gd32 + plecs` 双环境支撑了实机与仿真的协同开发

因此，这套代码非常适合作为数字电源项目的基础母版，用于后续派生 PFC、电池充电、逆变、双向变换、AC-DC、DC-DC 等不同类型的产品工程。
